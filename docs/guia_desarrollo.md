# Guía de Desarrollo

## Filosofía

Cada sprint debe dejar el programa completamente funcional.

## Reglas

- No romper compatibilidad hacia atrás.
- No usar referencias de celdas fijas.
- Todo el motor debe trabajar mediante tablas estructuradas.
- Todo cambio relevante debe quedar documentado.
- Todo cambio debe pasar pruebas antes de publicarse.

## Flujo de trabajo

Sprint → Implementación → Pruebas → Documentación → Commit → Release
